# Session model

purify uses sessions to accumulate conversation context across multiple tool calls. A session represents a single coherent interaction — the user provides a spec, the model asks clarifying questions, the user responds, and eventually the session produces an output.

Sessions are created when the first purify tool call arrives. Each subsequent call in the same session appends to the conversation history. The session knows what mode was requested, what the original input was, and the full message exchange so far.

Sessions need to expire. If a session hasn't had activity for some amount of time (maybe 30 minutes, should be configurable), it should be cleaned up. Stale sessions shouldn't sit around forever consuming memory.

What gets stored per session: the original input text, the mode, the conversation history (roles + content), the AISP if one has been generated, and some metadata like when it was created and last touched.

Session IDs are generated by purify. The client doesn't pick them. The ID is returned on session creation and the client passes it back on subsequent calls.

There's an open question about persistence. For now sessions can be in-memory. If the process restarts, sessions are lost. That's acceptable for the current use case. We can revisit if server mode becomes a thing.

When a session produces its final output, it stays accessible for a short window in case the client needs to retrieve something, then it can be cleaned up.
